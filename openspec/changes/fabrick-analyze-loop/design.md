## Context

Standalone Node.js CLI in `applications/loop/`. Runs outside of any user repo — operates on cloned corpus repos. Uses Claude API directly (Batch API for eval calls). Reads and writes `SKILL.md` in this repo.

Prerequisite: `fabrick-analyze-v2` merged. Loop starts with improved baseline skill.

## Goals / Non-Goals

**Goals:**
- Manage corpus of diverse real repos
- Run fabrick-analyze against each repo, store outputs
- Evaluate output quality via Q&A accuracy metric
- Improve SKILL.md iteratively using Opus, section by section
- Protect against regressions: reject patches that drop any metric >5%
- Stop when improvement plateaus

**Non-Goals:**
- Real-time interactive improvement (async batch loop)
- Automatic GitHub discovery (corpus curated manually for now)
- Full CI integration (run manually)
- Publishing improved SKILL.md anywhere (local git commit only)

## Architecture

```
applications/loop/
  corpus.yaml              # list of repo URLs
  scripts/
    run.js                 # clone + analyze + store
    evaluate.js            # Q&A scoring via Batch API
    improve.js             # patch SKILL.md via Opus + regression check
    loop.js                # orchestrator: run → eval → improve → repeat
  benchmarks/
    history.yaml           # metric trend across all runs
    <run-id>/
      <repo-name>/
        output/            # .fabrick/context/ copy
        qa_pairs.yaml      # ground truth Q&A
        scores.yaml        # per-category scores
      summary.yaml         # aggregate + delta vs prev run
```

## Decisions

### Iteration strategy: batch of 3–5 repos

Not per-repo improvement — accumulate failures across a batch before calling Opus. Avoids overfitting to single-repo edge cases. Single improvement cycle per iteration:

```
repos [1-5]   → run → eval → collect failures → improve → regression [1-5]   → commit if pass
repos [6-10]  → run → eval → collect failures → improve → regression [1-10]  → commit if pass
...
```

### Q&A metric

Ground truth generated from raw code (before analysis). Evaluation reads only `.fabrick/context/` output.

Question categories:
- **technical** (40%): "What message broker?", "What DB?", "What env vars required?"
- **product** (35%): "What is token TTL?", "What currencies supported?", "What happens on payment failure?"
- **architectural** (25%): "What events published?", "What services consumed?", "What protocols used?"

Score per category = % questions answered correctly. Overall = weighted average.

Questions auto-generated by Sonnet from raw code. No manual curation.

### Batch API usage

All Sonnet eval calls (Q&A generation + Q&A answering from context) submitted as Batch API requests. Async — loop waits for batch completion before proceeding to improve step. ~50% cost reduction vs synchronous calls.

Opus improve call is synchronous (one call, needs result immediately).

### Patch strategy: section-level, not full rewrite

Opus receives:
1. Current section from SKILL.md (e.g. "### 1b. Endpoints")
2. Top 5 failure cases from batch (question + expected answer + what context produced)
3. Instruction: return patched section only

Avoids regression from unrelated section changes. Section boundaries defined by `###` headings in SKILL.md.

### Regression check

After patch applied, re-run full corpus (all repos seen so far) and compare to previous run's scores.yaml per repo. Reject if any category drops >5% on any repo. Up to 3 retry attempts with failure context added. After 3 failures → log as hard conflict, skip section this iteration.

### Plateau detection

If overall weighted score improves <2% across a full iteration (all 3–5 repos + improve cycle), loop terminates.

### Corpus format

```yaml
repos:
  - url: https://github.com/...
  - url: https://github.com/...
```

No metadata. Skill must detect everything from code. This is intentional — tests generalization.

### Benchmark run ID

Format: `YYYY-MM-DD-NNN` (sequential per day). Stored in `benchmarks/` directory alongside `history.yaml`.

## Risks / Trade-offs

- Q&A ground truth quality depends on Sonnet reading raw code — if code is unclear, questions may be low-quality. Acceptable for bootstrap.
- Batch API latency (up to 24h) means loop isn't fast. Each iteration = one batch cycle. Plan for multi-day runs.
- Opus patches may improve one language's extraction while breaking another. Regression check catches this but may block progress on cross-language sections — surface as hard conflict.
- Corpus diversity is critical. Low diversity → skill overfits. Start with 10 repos across 5+ different stacks.
